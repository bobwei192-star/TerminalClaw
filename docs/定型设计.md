TerminalClaw 设计文档
基于 Tmux + OpenClaw + Ollama 的完全免费、本地化的终端日志 AI 分析方案

一、项目背景与目标
1.1 问题定义
开发者在终端中进行日志分析时面临两个核心痛点：

上下文割裂：看日志在左边，想用 AI 分析需要手动复制粘贴到浏览器或独立应用

成本门槛：现有终端 AI 方案（Warp Oz Agent、Cursor Terminal）均为付费订阅

1.2 解决方案
TerminalClaw 提供一套完全免费、完全本地、开箱即用的终端内 AI 日志分析工作流：左面板实时输出日志，右面板 OpenClaw Agent 就绪分析，通过本地文件实现数据桥接。

全本地	所有数据处理和模型推理均在本地完成
可封装	可打包为 npm 包或 CLI，一键分发

2.2 最终选择：方案B（Tmux）
核心决策理由：

Tmux 的脚本化能力是封装基础：send-keys 可以精确控制每个面板的行为，这是将整个工作流封装为 npm run dev:ai 命令的前提。Kitty 的 kitten @ 类似但文档和社区案例远不如 Tmux 丰富。

会话持久化是刚需：日志分析场景中，用户可能需要让日志持续运行，自己断开离开，回来再接入。Tmux 的 detach/attach 原生支持这一模式，Kitty 需要额外配置。

WSL2 兼容性：目标用户可能在 Windows 上开发，Tmux 在 WSL2 中运行最稳定，是近年来 WSL2 环境下的标准终端复用器。

截至 2026 年 5 月，OpenClaw 各版本的已知 Bug 及其修复状态如下。TerminalClaw 的设计方案依据这些状态进行动态调整。

4.3.2 关键发现与策略调整

（1）read / write 工具：✅ v2026.4.5 已修复

这是对 TerminalClaw 最重要的修复。v2026.4.5 的 changelog 明确记录了对大文件 read 输出的溢出保护机制。这意味着 Agent 现在可以直接使用 read 工具读取日志文件，不再需要依赖 exec cat 的迂回方式。对于 TerminalClaw 而言，日志分析场景的核心操作——读取日志文件——现在有了更直接、更可靠的实现路径。

（2）Windows 原生 exec 空输出：❌ 仍未修复

GitHub Issue #19631 和 #18965 确认，从 v2026.2.15 引入的回归至今未修复。降级到 v2026.2.9 可立即恢复正常的 exec 行为。结论不变：Windows 原生环境 OpenClaw 不可用，TerminalClaw 必须要求 WSL2 环境。

（3）v2026.4.5 引入新回归：❌ Docker 中 exec 静默失败

GitHub Issue #62115（S 级权威来源）报告，从 v2026.4.1 升级到 v2026.4.5 后，Docker 部署中的 exec 工具静默失败，即使开启了 security: 'full' 和 ask: 'off'，命令也返回空字符串。对于 Docker 部署用户，需降级到 v2026.4.1，或改用 npm 原生安装。

（4）Ollama 上下文窗口溢出：⚠️ 持续存在

Issue #59957 确认，即使配置了较小的上下文窗口，OpenClaw v2026.3.13+ 仍会向 Ollama 发送约 262,144 (265K) 的超大上下文窗口。绕过方案是在 openclaw.json 中使用 contextTokens 覆盖（而非仅 contextWindow），强制将上下文限制在合理范围（如 8192 tokens）。

4.3.3 更新后的核心策略

版本选择：非 Docker 部署推荐 v2026.4.5+（read 已修复，直接使用 read 读取日志）；Docker 部署推荐 v2026.4.1（避免 exec 静默失败回归）。

工具选择：现在可以放心使用 read 工具读取日志文件，这是更自然、更语义化的方式。exec 仍作为补充，用于 grep、awk 等复杂文本处理。

Ollama 配置：必须同时配置 contextTokens 和 contextWindow，以强制覆盖 OpenClaw 的默认超大上下文窗口行为。

环境约束：Windows 用户必须使用 WSL2，这一结论未变。

五、技术规格与配置
5.1 环境要求

TerminalClaw 的设计目标是在主流开发环境中零配置运行。以下是各层级的兼容性设计论证：

操作系统层：Linux（Ubuntu 22.04+、Debian 12+）和 macOS（12+，Apple Silicon 推荐）为 Tier-1 原生支持。Windows 用户必须通过 WSL2 使用 Ubuntu 22.04+ 环境——这是 OpenClaw 在 Windows 原生环境存在 exec 空输出 Bug（未修复）的硬性约束，而非 TerminalClaw 的设计选择。

运行时层：Node.js 20.0.0+（推荐 24.x）用于控制脚本的运行；Tmux 3.3+ 提供分屏与会话持久化；Ollama 0.3.0+ 提供本地模型推理。OpenClaw 的版本选择需根据部署方式决定——非 Docker 部署推荐 v2026.4.5+（read 工具已修复），Docker 部署推荐 v2026.4.1（v2026.4.5 存在 Docker 中 exec 静默失败的回归）。

硬件层：推荐 4 核以上 CPU、16GB 以上内存、10GB 以上磁盘空间（用于模型下载与日志缓存）。8GB 内存虽可运行但模型推理响应较慢，影响分析体验。

5.2 OpenClaw 配置设计

OpenClaw 的配置文件位于 ~/.openclaw/openclaw.json，TerminalClaw 对其有以下设计约束：

Provider 配置：使用 Ollama 作为本地 Provider，API 地址指向本地 127.0.0.1:11434（Ollama 默认端口）。模型使用 custom Modelfile 创建的 terminalclaw 专用模型，以 qwen3.5 作为后备。

上下文窗口配置（关键）：由于 OpenClaw v2026.3.13+ 存在 Ollama 上下文窗口强制 265K 的 Bug，配置文件必须同时设置 contextTokens 和 contextWindow 两个参数进行双重覆盖。contextTokens 用于强制限制实际发送的 token 数量（推荐 8192），contextWindow 用于声明模型能力上限（推荐 65536）。仅配置 contextWindow 不足以绕过该 Bug，这是基于社区验证的结论。

工具权限设计：exec 工具启用 sandbox 模式，白名单限制为与日志分析相关的安全命令——文件读取类（cat、tail）、文本搜索类（grep）、JSON 处理类（jq）、文本处理类（awk、sed），且限定操作目录为 /tmp/tclaw/。read 工具在 v2026.4.5+ 中已修复，可直接用于读取日志文件，作为 exec cat 的更优替代。

其他配置：memory 启用本地持久化存储，使 Agent 在多轮对话中保持上下文记忆。telemetry 关闭，确保完全的本地化与隐私。

5.3 Ollama 模型配置设计

基础模型选择 qwen3.5，这是一个在中文技术文本理解方面表现优异的 8B 级开源模型。通过 Ollama 的 Modelfile 机制创建自定义模型 terminalclaw，关键参数设计如下：

上下文窗口（num_ctx）：设置为 65536（64K），允许 Agent 一次性处理大量日志内容。这是日志分析场景的核心参数——过小则需频繁截断，过大则增加推理延迟。

温度（temperature）：0.3，偏向确定性输出。日志分析不是创意写作，需要准确、一致的分析结果。较低的 temperature 减少幻觉，提高分析可靠性。

Top-P 采样：0.9，在确定性分析基础上保留适度的灵活性，使 Agent 在措辞和解释时不过于机械。

重复惩罚（repeat_penalty）：1.1，避免在长分析中重复相同的措辞模式。

系统提示词设计：定义 Agent 的角色（终端日志分析专家）、核心能力（多类型日志读取、异常识别、根因分析、可执行修复建议）、工作方式（优先使用 read 工具读取日志文件，exec grep/awk 辅助，分析结果含问题概要、根因、影响范围、修复建议，使用中文回复保留英文术语）、安全约束（只分析不修改、不确定时标注推测、不执行危险命令）。

5.4 Tmux 配置设计

TerminalClaw 推荐的 Tmux 配置围绕提升日志分析体验展开：

鼠标支持：开启鼠标模式，允许用户点击切换面板、滚动查看日志历史，降低纯键盘操作的学习门槛。

历史缓冲区：设置为 50000 行，确保用户在长时间运行后仍可向上滚动查看历史日志输出。这一容量对于典型的开发调试会话（数小时内）足够覆盖。

面板状态栏：顶部显示面板编号和自定义标题（如"📋 日志输出"和"🤖 OpenClaw AI"），帮助用户快速识别左右面板的功能角色。

状态栏位置：底部显示 Tmux 会话状态，保持与标准终端体验一致。

面板索引：统一从 1 开始编号（而非默认的 0），降低认知负担。

快捷键设计：将默认前缀从 Ctrl+B 改为 Ctrl+A（GNU Screen 用户习惯），垂直分屏绑定为 | 键（左/右面板），水平分屏绑定为 - 键（上/下面板）。这一设计兼顾了 GNU Screen 迁移用户的肌肉记忆和新用户的直觉理解。
六、核心脚本设计

6.1 主控脚本：terminalclaw.sh

主控脚本是整个 TerminalClaw 工作流的编排中枢，采用 Bash 实现，依赖 tmux、openclaw、ollama 三个组件。脚本遵循 MIT 许可发布。

6.1.1 配置体系设计

脚本采用环境变量驱动的配置模式，所有关键路径和参数均可通过环境变量覆盖，无需修改脚本代码：

日志缓存路径：默认 /tmp/tclaw，可通过 TCLAW_LOG_DIR 环境变量重定向。选择 /tmp 目录的理由是其作为内存文件系统（tmpfs）挂载时读取速度极快，且系统重启后自动清理，避免日志残留。

日志文件名：固定为 live.log，位于日志缓存目录下。Agent 通过固定路径获取最新日志，无需动态探测。

Tmux 会话名：默认 terminalclaw，可通过 TCLAW_SESSION 自定义。支持同一用户运行多个独立 TerminalClaw 实例。

日志文件轮转阈值：内置 500MB 上限。当日志文件超过此值，自动将当前文件重命名为带时间戳的归档文件，并清空 live.log 继续写入。这一设计在"不丢日志"和"不占满磁盘"之间取得平衡。

模型名：默认 terminalclaw，通过 TCLAW_MODEL 环境变量可切换为其他 Ollama 模型（如直接使用 qwen3.5）。

6.1.2 颜色输出系统

脚本内建 ANSI 颜色码输出，将日志信息按级别分类——INFO（绿色）表示正常状态、WARN（黄色）表示需关注但可继续、ERROR（红色）表示阻断性问题。这一设计使脚本在终端中的输出与日志分析面板中的输出保持一致的视觉层次。

6.1.3 环境自检模块

环境自检是脚本启动时的第一道关卡，按以下顺序逐项验证：

（1）二进制存在性检查：依次验证 tmux、openclaw、ollama、tee 四个命令是否在 PATH 中可找到。任一缺失则输出安装指令并终止。

（2）Ollama 服务可用性：通过 HTTP 请求 http://127.0.0.1:11434/api/tags 检测 Ollama 服务是否在运行。这是比进程检测更可靠的判断方式——服务可能已启动但 API 尚未就绪。

（3）模型就绪性：检查指定模型是否已存在于 Ollama 模型中。若不存在，尝试通过 Modelfile 自动创建；若 Modelfile 也不存在，自动拉取 qwen3.5 作为后备。这一优雅降级的设计确保了用户无需手动准备模型。

设计要点：自检模块将所有验证集中在一处，任一失败即终止并给出明确指引。这避免了启动后才发现依赖缺失导致的面板创建失败或 Agent 无法工作。

6.1.4 日志文件生命周期管理

初始化（setup）：创建日志目录（如不存在），touch 空日志文件以确保后续写入路径畅通。

轮转（check）：每次 monitor 心跳和 start 启动时检查文件大小。超过 500MB 阈值时，将当前日志文件重命名为 live.log.<Unix时间戳>.old，然后创建新的空日志文件。旧文件保留在磁盘上供事后回溯分析，用户可手动清理。

设计考量：500MB 阈值基于典型日志场景估算——以每行 200 字节计算，约 250 万行，足以覆盖数小时的高强度日志输出。阈值可通过修改脚本常量调整。

6.1.5 Tmux 会话编排

Tmux 会话管理是终端交互的核心，分为以下几个设计层次：

会话存在性判断：通过 tmux has-session 检测，避免重复创建同名会话导致资源冲突。

工作区创建（launch）：使用 tmux new-session -d 创建后台会话（-d 表示不立即 attach），然后 split-window -h 进行垂直分屏。面板命名"📋 日志输出"和"🤖 OpenClaw AI"通过 pane-border-format 显示在状态栏中，帮助用户快速识别。

面板间命令注入（send-keys）：通过 tmux send-keys 向指定面板发送按键序列，实现自动化的面板初始化。这是 Tmux 脚本化的核心能力——send-keys 可以精确控制每个面板中执行的命令，无需用户手动操作。

左面板初始化流程：先输出欢迎横幅和日志文件路径提示，然后以 eval 方式执行用户提供的日志命令并通过 tee -a 追加写入日志文件。eval 的设计是为了支持管道和重定向等复杂 shell 语法。

右面板初始化流程：输出可用指令列表（分析最近50行、检查错误、分析异常模式等），然后启动 OpenClaw Agent 并附带系统提示词。提示词中告知 Agent 日志文件路径和基本工作方式（优先使用 read 工具读取日志文件，必要时使用 exec grep/awk 进行复杂文本处理）。

会话接入（attach/detach）：用户可通过 tclaw attach 随时接入已有会话，通过 Ctrl+B D 断开离开。后台日志持续写入，Agent 状态保持，下次接入时无缝恢复。

6.1.6 命令处理架构

脚本采用经典的"命令-子命令"模式，通过 case 分支路由到各处理函数：

start：核心启动命令。执行顺序为 环境自检 → 日志目录初始化 → 会话存在性检查 → 工作区创建 → 面板初始化 → 附加到会话。每个步骤失败均有明确错误提示和退出码。

attach：接入已有会话。仅执行会话存在性检查，存在则 attach，不存在则提示使用 start 创建。

stop：停止工作区。使用 tmux kill-session 注销会话，日志文件保留在磁盘上不删除，供事后分析。

analyze：独立分析模式。无需启动完整 Tmux 会话，直接调用 OpenClaw Agent 读取日志文件并分析最近 N 行（默认 100 行，可参数指定）。适用于已有运行中的日志但不想进入多面板界面的场景。

monitor：后台监控模式。进入无限循环，每 30 秒执行一次日志文件大小检查和错误计数扫描。错误计数通过 grep -ci 统计 ERROR/FATAL/EXCEPTION 行数实现，非零时触发 Agent 快速诊断并输出结果。此模式不依赖 Tmux，可在后台运行（使用 & 或 systemd 管理）。

status：状态查看。输出 Tmux 会话状态（运行中/未运行，面板列表）、日志文件信息（大小、行数）、Ollama 服务状态。一站式的健康检查。

help：内置帮助信息。包含完整命令列表、使用示例、环境变量说明，使脚本自文档化。

6.1.7 入口设计

main 函数作为脚本唯一入口，接收第一个参数作为命令名（默认 help），通过 case 分支路由。使用 set -euo pipefail 启用严格错误处理模式，确保任何管道失败或未定义变量引用都会终止脚本，避免静默失败。

6.2 封装为 npm 包

TerminalClaw 通过 npm 包发布，实现一键全局安装（npm install -g terminalclaw）后即可使用 tclaw 命令。

包结构设计：

bin 入口：通过 package.json 的 bin 字段注册两个命令——tclaw（Node.js 包装器，负责跨平台路径解析和环境变量注入）和 terminalclaw（Bash 主控脚本本体）。双入口设计兼顾了 npm 包规范和实际脚本的独立性。

npm scripts 集成：提供 dev:ai、dev:monitor、dev:analyze 三个便捷脚本，使项目开发者可以在 package.json 中直接引用 TerminalClaw 工作流，无需记忆完整命令。

files 发布策略：仅发布 bin/、scripts/、config/ 三个目录，排除测试文件、文档和开发依赖，保持包体积最小。

关键词优化：使用 terminal、ai、log-analysis、openclaw、ollama、tmux 六个关键词，覆盖用户在 npm 搜索时可能使用的各类术语。

八、已知限制与风险

8.1 技术限制

非真正流式分析：OpenClaw 是"请求-响应"模式，无法像 Warp Oz Agent 那样 Attach 到 PTY 缓冲区持续分析。TerminalClaw 通过后台轮询（monitor 模式）逼近实时效果，但延迟在 30 秒级别。这不是 Bug，而是 OpenClaw 工具模型的架构特性。

模型性能上限：本地 8B 模型（qwen3.5）在分析复杂堆栈跟踪时不如云端大模型（GPT-4/Claude），但对常规日志分析场景（错误定位、异常模式识别、根因推断）足够用。随着 Ollama 支持的模型持续进化，未来可通过切换更大模型提升分析质量。

WSL2 强制：OpenClaw 在 Windows 原生环境存在 exec 空输出 Bug（自 v2026.2.15 起未修复），TerminalClaw 必须要求 WSL2 环境。这不是方案自身的设计限制，而是 OpenClaw 上游的环境约束。

9.2 "终端 + OpenClaw" 组合方案

终端 + OpenClaw 是一个已被社区验证的方向，以下项目从不同角度切入这一领域。值得注意的是，这些项目没有一个是针对"通用日志分析 + 全本地化"这一具体组合的。

（1）TroyKelly OpenClaw TMux Session Management

这是 GitHub 上 openclaw-projects 的 Epic #1667，是目前最完整的 "OpenClaw + tmux + 终端" 集成方案。核心设计包含：Agent 可通过 7 个 plugin 工具（terminal_session_start、terminal_send_command、terminal_capture_pane 等）管理 tmux 会话；基于 xterm.js 的 Web 终端前端，支持实时 I/O、窗口标签页切换、全屏模式；Worker（gRPC）→ API（REST/WebSocket）→ 前端 三层架构。

与 TerminalClaw 的关键差异：TroyKelly 的方案是通用远程终端管理平台——Agent 可以操作任意 SSH 连接上的 tmux 会话，需要 Web 前端介入，日志分析只是它可能被使用的场景之一。TerminalClaw 聚焦于单一场景（日志分析），无需浏览器、无需 API 层，直接在终端内完成全部操作。两者方向互补而非竞争。

（2）Oh-My-OpenClaw

happycastle114 维护的 OpenClaw 插件框架。其 Agent 会话持久化层完全基于 tmux 实现——使用共享 socket 目录 `/tmp/openclaw-tmux-sockets/openclaw.sock`，Agent 在独立 tmux 窗口中运行，断开终端不会终止，支持 send-keys 注入和 detach/attach 会话保持。

与 TerminalClaw 的相似点：都用 tmux 做持久化、都用 send-keys 控制面板。差异在于 Oh-My-OpenClaw 是通用 Agent 编排平台（多 Agent 管理、多 Skill 调度），TerminalClaw 是为日志分析场景从零搭建的专用工作流。两者的 tmux 用法设计可互相参考。

（3）OpenClaw Tmux Skill（官方）

OpenClaw 官方发布的 `@openclaw/openclaw` tmux Skill（2026-02-02），是三者中最轻量的方案——仅一个 Skill 文件即可让 Agent 获得远程控制 tmux 会话的能力。被社区编辑器评为"将 AI 助手转变为强大终端操作员"的高分 Skill。

差异：这只是一个 Skill（让 Agent 学会操作 tmux），不提供面板布局预设、日志桥接管道、工作流封装等完整方案。它解决的是"Agent 能不能操作 tmux"，TerminalClaw 解决的是"操作 tmux 来做什么"——这是能力层和场景层的区别。

（4）Security Log Analyzer Skill

OpenClaw Skills 生态中的专用日志分析 Skill。功能为：用户提供日志文件路径 → Agent 读取 → LLM 分析 → 输出结构化报告（含威胁等级、攻击链、IOC 提取）。支持简要分析和详细鉴证两种模式，内置 token 截断保护（>4000 tokens 自动截断）和 API 限流重试（429 错误 10 秒自动重试）。

差异：功能层面最接近 TerminalClaw（都是日志分析），但两个关键分叉——它是安全日志专用（非通用应用日志），且依赖云端 API（SiliconFlow）而非本地模型。TerminalClaw 定位在"开发者的日常日志分析"，用本地 Ollama 实现零外部依赖。

（5）OpenClaw Logger

massaindustries 维护的 Web Dashboard，用于实时追踪 OpenClaw Agent 的会话行为。提供每一条消息和工具调用的实时可视化、本地处理、行为模式分析、AI 辅助调试。创建于 2026-02-17（7 commits），尚处早期。

差异：OpenClaw Logger 监控的是 Agent 自身的行为（Agent 发了什么消息、调了什么工具），而 TerminalClaw 分析的是应用日志。两者分析对象不同，无竞争关系。

9.3 差异化总结

"终端 + OpenClaw" 方向已被多个独立方案验证（TroyKelly 的 Epic、tmux Skill、Oh-My-OpenClaw），但没有一个方案同时满足以下组合：

维度	TroyKelly	Oh-My-OpenClaw	tmux Skill	Security Log Analyzer	OpenClaw Logger	TerminalClaw

通用日志分析	❌ 通用终端	❌ 通用 Agent	❌ 通用终端	⚠️ 仅安全日志	❌ Agent 监控	✅ 通用日志
全本地 Ollama	❌ 依赖 API	❌ 依赖 API	❌ 依赖 API	❌ 云端 API	❌ 依赖 API	✅ 零外部依赖
一键封装	❌ 需多步配置	❌ 需 npm 安装	⚠️ 需手动加载	❌ 需 Skill 配置	❌ 需 Docker	✅ npm 一键

TerminalClaw 的组合（终端内双面板 + 通用日志分析 + 全本地 Ollama + 一键封装 + 完全免费）在现有生态中是独特的。

9.4 最值得借鉴的 GitHub 项目

在调研过程中，以下三个项目在功能定位或架构设计上与 TerminalClaw 存在高度相关性，值得在设计实现时深入借鉴。

（1）Gonzo —— 最接近的日志分析 TUI ⭐⭐⭐⭐⭐

仓库：github.com/control-theory/gonzo | 语言：Go | 许可：MIT | 90+ commits

Gonzo 是一个基于 Go 开发的终端日志分析 TUI 工具，灵感来源于 k9s。其核心能力与 TerminalClaw 高度重叠：

实时日志处理：支持 stdin、文件、OTLP 网络传输三种输入源，自动检测 JSON、logfmt 和纯文本格式。这与 TerminalClaw 的"tee 中转 → 文件中转"方案形成对比——Gonzo 走的是原生管道流，TerminalClaw 走的是文件持久化。

2×2 网格仪表盘：包含实时图表、日志视图和统计信息三个区域，Vim 风格键盘导航 + 鼠标支持。TerminalClaw 的区别在于采用 Tmux 双面板分离——左侧（日志源）和右侧（AI 分析）是两个独立的终端进程，而非一个 TUI 内部的分区。

AI 分析集成：内置 AI 分析功能，支持 OpenAI、Ollama 等多种模型，可以完全离线运行。用户选中一条日志后按键即可触发 AI 分析。TerminalClaw 的 AI 交互方式是对话式的（OpenClaw Agent 常驻），Gonzo 是点击触发式的。

借鉴要点：Gonzo 的 AI 交互模式证明"选中日志 → 一键 AI 分析"是用户刚需；其 Ollama 集成方式（OPENAI_API_KEY=ollama + OPENAI_API_BASE=http://localhost:11434）是本地模型接入的成熟模式；Severity 过滤的交互设计（Ctrl+f 弹出模态窗口，颜色编码，实时计数）可作为 TerminalClaw 增强面板的参考。

与 TerminalClaw 的关系：Gonzo 是 TUI 单体（所有功能在一个进程内），TerminalClaw 是多进程组合（Tmux + Bash + OpenClaw Agent）。两者处理同一场景，但架构哲学不同——前者追求"一体化 TUI 体验"，后者追求"模块解耦 + 脚本化封装"。不存在直接竞争，可以互相参考。

（2）Claw Core —— OpenClaw 插件化的最佳参考 ⭐⭐⭐⭐

npm：@wchklaus97hk/claw-core | 许可：MIT | 9 个版本

Claw Core 是 OpenClaw 生态中目前结构最完整的 Terminal Runtime Layer 插件，通过 openclaw plugins install @wchklaus97hk/claw-core 即可安装。其架构设计对 TerminalClaw 的插件化演进有直接参考价值：

插件化架构：Claw Core 将终端运行时能力（daemon 守护、多机器人管理、Agent Teams、Cursor Agent 直调）全部封装为 OpenClaw Plugin，而非独立脚本。这意味着它可以直接被 OpenClaw Gateway 发现、加载和执行，无需用户手动调用外部脚本。

daemon 管理：提供 clawcore start/stop/status 三个命令，自动下载插件二进制文件并在首次启动时自动配置 openclaw.json。这一设计模式与 TerminalClaw 的 tclaw start/stop/status 命令体系高度对应，区别在于 Claw Core 运行在 OpenClaw 内部，TerminalClaw 作为独立 Bash 脚本运行在 OpenClaw 外部。

Agent Teams：支持多 Agent 协作，使用共享任务面板和 Telegram 群聊协作。TerminalClaw 的 v1.0 路线图中也规划了多 Agent 协作，可以参考 Claw Core 的 Agent 间消息路由、任务分派和工作区隔离机制。

局限性：该项目自述"尚未完全集成"、"可能遇到边缘情况"，声明"生产使用需谨慎"。这表明将终端运行时能力封装为 OpenClaw Plugin 在稳定性上仍有挑战，TerminalClaw 当前采用独立脚本的方案在可靠性上更保守。

借鉴要点：如果 TerminalClaw 未来要做成 OpenClaw Plugin，Claw Core 的 api.registerTool() 模式和 SKILL.md 结构是最直接的学习样本。近期的路线是保持独立脚本形态确保稳定性，远期可考虑两种形态并存（独立 CLI + Plugin）。

（3）rejoin —— AI Agent 会话索引与 Tmux 恢复 ⭐⭐⭐⭐⭐

PyPI：rejoin | 语言：Python | 许可：MIT

rejoin 是一个本地 AI Agent 会话管理仪表盘，统一索引和管理 Claude Code、Codex、OpenClaw、Hermes 等 6 个不同 AI Agent 的会话。它在架构上与 TerminalClaw 高度互补：

OpenClaw 会话文件读取：直接读取 ~/.openclaw/agents/**/*.jsonl 的会话文件，证明了"读取 OpenClaw 本地会话文件"这条路是通的。TerminalClaw 可以让 Agent 分析历史记录保存到相同格式，使 rejoin（或其他类似工具）能索引 TerminalClaw 的分析输出。

Tmux 原生集成：在 Tmux 内部按回车键直接在新窗口恢复历史会话，零摩擦体验。这与 TerminalClaw 的 tclaw attach 类似，但 rejoin 解决的是"恢复历史 Agent 对话"，TerminalClaw 解决的是"恢复正在运行的日志分析会话"。

终端 + Web 双界面：基于 Textual 框架的 TUI + Web UI。TerminalClaw 目前只有终端界面，未来如果需要 Web 远程查看分析结果，可以参考这种双界面架构。

借鉴要点：重点研究它的 OpenClaw .jsonl 文件解析逻辑——如何索引会话、如何提取关键元数据（时间、模型、token 消耗、工具调用记录）。TerminalClaw 的 analyze 和 monitor 命令可以输出结构化的分析记录，让 rejoin 这类工具能索引 TerminalClaw 的历史分析，形成工具链联动。

十一、发布为 OpenClaw Agent 或 Skill 的可行性分析

TerminalClaw 当前采用独立 Bash 脚本 + npm 包的形态运行在 OpenClaw 外部。本节分析将其发布为 OpenClaw 生态内的一等公民（Agent 或 Skill）的可行性与路径。

11.1 OpenClaw 生态的两种分发形态

OpenClaw 提供了两种将自定义功能融入其生态的方式：

形态	定义	分发方式	适用场景
Agent	一个持久的 AI 助手实例，有自己的工作区、记忆、技能集和渠道绑定	openclaw agent 注册	需要持续运行、独立记忆、多轮对话的复杂工作流
Skill	一个可插拔的功能模块，以 Markdown 指令或脚本定义新的工具和能力	ClawHub 发布，openclaw plugins install	单一功能增强、工具注册、知识注入

TerminalClaw 的定位介于两者之间——它有完整的会话生命周期管理（偏向 Agent），但功能聚焦于日志分析这一个垂直场景（偏向 Skill）。

11.2 作为 OpenClaw Skill 发布：最可行路径

这是 TerminalClaw 进入 OpenClaw 生态的最短路径。可行性分析如下：

优势（高可行性）：

Skill 的安装门槛最低：用户只需在 OpenClaw 中执行一次安装命令，无需额外学习独立的 CLI 工具。参考 Claw Core 的发布路径（npm 发布 + openclaw plugins install），TerminalClaw 完全具备走这条路的条件。

Skill 只需定义功能说明和工具注册：TerminalClaw 的核心能力（启动 Tmux 面板、读取日志文件、调用 Ollama 分析）可以通过 SKILL.md 描述 + 注册自定义工具的方式暴露给 Agent。Agent 可以在对话中自动调用这些工具。

与现有官方 tmux Skill 可以叠加使用：OpenClaw 官方已经发布的 @openclaw/openclaw tmux Skill 提供了底层的 Tmux 操作能力（send-keys、捕获面板输出），TerminalClaw Skill 可以在此基础上提供日志分析场景的编排逻辑，形成"基础能力层 + 场景编排层"的分层架构。

挑战（需要解决）：

Bash 脚本无法直接嵌入 Skill：Skill 定义文件（SKILL.md）本质是 Markdown 格式的指令文档，不执行外部脚本。TerminalClaw 的启动逻辑（tclaw start 中涉及 Tmux 会话创建、面板初始化、环境检测）需要通过 OpenClaw 的工具调用机制间接实现，而不能直接执行 Bash 脚本。

环境依赖验证的责任转移：当前 TerminalClaw 的环境自检在 Bash 脚本中完成。作为 Skill 发布后，需要依赖 OpenClaw 的 exec 工具来执行环境检测命令（如 which tmux、curl http://localhost:11434/api/tags），这引入了额外的消息往返延迟。

Tmux 面板布局的自动化创建：在独立脚本中通过 send-keys 精确控制面板是最自然的实现。作为 Skill 则需要告诉 Agent"创建名为 terminalclaw 的 Tmux 会话，垂直分屏，左面板执行日志命令并 tee 到文件，右面板启动 OpenClaw 并等待分析指令"，让 Agent 自行编排。这依赖 Agent 对 Tmux 的能力理解是否准确。

推荐路径：

近期（v0.1-v0.2）：保持独立 CLI 形态。这是风险最低、实现最快的方式，用户可以立即使用 tclaw start 启动完整工作流。

中期（v0.3）：发布 TerminalClaw Skill。将核心能力拆分为多个工具（terminalclaw_start、terminalclaw_analyze、terminalclaw_status），通过 SKILL.md 注册到 OpenClaw。用户可以在已有的 OpenClaw Agent 对话中直接请求"启动 TerminalClaw 日志分析"。

最终态（v1.0）：两种形态并存。独立 CLI 面向喜欢脚本化操作的开发者，OpenClaw Skill 面向已经深度使用 OpenClaw 生态的用户——两者共享同一套核心脚本逻辑。

11.3 作为 OpenClaw Agent 发布：远期可能性

将 TerminalClaw 作为独立 Agent 发布意味着它成为 OpenClaw 中的一个"专职日志分析助手"，拥有自己的工作区、记忆和技能集。可行性分析如下：

优势：Agent 形态可以实现最自然的交互——用户通过 Telegram/Slack/WhatsApp 发送日志分析请求，Agent 自动执行分析并返回结果。结合 monitor 模式，Agent 可以主动推送告警。

挑战：Agent 的工作区管理、记忆持久化、渠道配置等基础设施依赖 OpenClaw Gateway 的运行，部署复杂度比独立 CLI 高一个数量级。对于"只想在终端里分析日志"的用户而言过重。

建议：v1.0 阶段考虑 Agent 形态的可行性。在此之前，Skill 形态足以覆盖绝大多数使用场景。

11.4 与现有 OpenClaw 生态的互补关系

发布形态	与哪个现有生态组件互补	互补方式

Skill	@openclaw/openclaw tmux Skill	TerminalClaw Skill 提供场景编排，tmux Skill 提供底层 Tmux 操作

附录 A：故障排除指南

TerminalClaw 内建了完善的环境自检机制（见 §6.1.3），大多数问题在启动阶段就会明确报错。以下按问题域分类说明诊断思路和解决路径：

Ollama 连接失败：首先确认 Ollama 服务进程在运行（ollama serve），然后通过 curl http://127.0.0.1:11434/api/tags 验证 API 端点可达。常见原因包括端口冲突（默认 11434）、防火墙拦截、或 Ollama 未安装。TerminalClaw 的 start 命令会在环境自检阶段自动执行此验证。

OpenClaw 命令返回空：分两种情况排查。非 Docker 部署确认版本为 v2026.4.5+（openclaw --version），旧版本存在 read/write 工具不执行的 Bug；Docker 部署需确认版本为 v2026.4.1（v2026.4.5 存在 Docker 中 exec 静默失败的回归）。跨平台用户（Windows）确认在 WSL2 环境中运行，而非 Windows 原生 PowerShell。

OpenClaw 上下文窗口异常大：检查 ~/.openclaw/openclaw.json 中是否同时配置了 contextTokens 和 contextWindow。仅配置 contextWindow 不足以绕过 v2026.3.13+ 的 Ollama 上下文溢出 Bug（强制 265K）。推荐 contextTokens 设置为 8192，contextWindow 设置为 65536。

Tmux 面板创建失败：通常由僵死会话导致。使用 tmux kill-server 清理所有会话后重试，或使用 tmux new -s test 验证 Tmux 基础功能是否正常。确认 Tmux 版本 ≥ 3.3（tmux -V）。

日志文件不更新：检查文件存在性（ls -la /tmp/tclaw/live.log）和写入权限。核心验证点是左面板中的日志命令是否包含 tee -a（append 模式），而非覆盖写入（>）。

工具突然丢失：这是 OpenClaw v2026.3.x 的已知偶发 Bug，v2026.4.5+ 已改善但非 100% 修复。临时绕过方案为重启 OpenClaw Gateway 服务或降级到 v2026.2.9。长期方案等待 OpenClaw 上游完全修复。
