TerminalClaw 架构设计文档
============================

面向运维/开发者的本地优先 AI 终端工作空间——基于 Tmux 的持久化分屏环境，集成 OpenClaw 生态，零 API 费用实时分析日志与排障。

## 一、项目背景与目标

### 1.1 问题定义
开发者在终端中进行日志分析时面临两个核心痛点：
- **上下文割裂**：看日志在左边，想用 AI 分析需要手动复制粘贴到浏览器或独立应用
- **成本门槛**：现有终端 AI 方案（Warp Oz Agent、Cursor Terminal）均为付费订阅

### 1.2 解决方案
TerminalClaw 提供一套完全免费、完全本地、开箱即用的终端内 AI 日志分析工作流：左面板实时输出日志，右面板 OpenClaw Agent 就绪分析，通过本地文件实现数据桥接。

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| 零成本 | 不依赖任何付费 API 或订阅服务 |
| 全本地 | 所有数据处理和模型推理均在本地完成 |
| 可封装 | 可打包为 npm 包或 CLI，一键分发 |
| 解耦架构 | 各组件独立，可替换、可升级、可降级 |

### 2.2 组件与职责

| 组件 | 职责 | 选型 | 许可证 |
|------|------|------|--------|
| 终端复用器 | 分屏管理、会话持久化 | Tmux | ISC（自由） |
| AI Agent 框架 | 工具编排、Skills 管理 | OpenClaw | MIT |
| 本地 LLM | 文本推理、日志分析 | Ollama + qwen3.5 | MIT |
| 数据桥接 | 日志实时写入、Agent 按需读取 | 本地文件系统 + tee | N/A |
| 控制脚本 | 一键启停、工作流编排 | Bash / Node.js CLI | MIT |

### 2.3 数据流

```
[应用进程] ──stdout──▶ [tee] ──┬──▶ [终端显示] (用户可视)
                              │
                              └──▶ [/tmp/tclaw/live.log] (持久化)
                                       │
                                       │ read / exec (按需)
                                       ▼
                              [OpenClaw Agent]
                                       │
                                       ▼
                              [Ollama 本地模型]
                                       │
                                       ▼
                              [分析结果] ──▶ [右侧面板输出]
```

**数据流特点：**
- **单向写入**：日志源只写入，不读取，无锁竞争
- **实时监听**：Live Monitor 守护进程持续监听日志文件变化，检测到错误自动分析
- **文件轮转**：超过阈值自动归档，防止磁盘占满

## 三、核心交互模型

### 3.1 用户体验流程

| 用户操作 | 系统行为 |
|----------|----------|
| 1. 终端内执行 `$ tclaw start` | 检测 Ollama 服务状态、检测 OpenClaw 安装、创建日志缓存目录、启动 Tmux 会话、初始化双面板布局 |
| 2. 左面板自动运行 `npm run dev` | 应用日志持续输出，同时 tee 到缓存文件 |
| 3. 右面板就绪 | 显示欢迎信息和可用命令提示 |
| 4. 用户输入分析指令 "分析最近50行ERROR" | Agent 理解意图、使用 read 工具读取日志、调用本地 LLM 分析、流式输出结果 |
| 5. 用户 detach 离开 `Ctrl+B D` | Tmux 会话后台保持，日志继续写入文件 |
| 6. 用户重新接入 `$ tclaw attach` | Tmux 会话恢复，面板状态完整还原 |

### 3.2 交互协议设计

TerminalClaw 采用实时 Attach 模式，右面板启动 Live Monitor 守护进程，持续监听日志文件变化：

| 用户指令 | 系统行为 | 触发方式 |
|----------|----------|----------|
| 分析最近50行 | 读取最后50行日志并分析 | 右面板中直接输入 |
| 检查错误 | 搜索 ERROR/FATAL 并分析 | 自动触发（Live Monitor） |
| 实时监控 | 每 3 秒扫描一次，有错即报 | 自动持续运行 |
| 分析接口性能 | 提取请求耗时统计 | 右面板中直接输入 |
| 定位原因 | 堆栈跟踪定位到代码行 | 自动触发（检测到 Traceback） |

## 四、技术规格与配置

### 4.1 环境要求

- **操作系统层**：Linux（Ubuntu 22.04+、Debian 12+）和 macOS（12+）为 Tier-1 原生支持。Windows 用户必须通过 WSL2 使用。
- **运行时层**：Node.js 20.0.0+、Tmux 3.3+、Ollama 0.3.0+（本地模式）或 DeepSeek API（云端模式）、OpenClaw v2026.4.5+
- **硬件层**：推荐 4 核以上 CPU、16GB 以上内存、10GB 以上磁盘空间

### 4.2 双模式架构

TerminalClaw 支持两种解释模式，通过 `TCLAW_INTERPRETER` 环境变量控制：

| 模式 | TCLAW_INTERPRETER | LLM 调用方式 | 延迟 | 工具链 | 推荐 |
|------|------------------|-------------|------|--------|------|
| API 直连 | `direct_api` (默认) | Python requests → DeepSeek | 3-5s | 无 | ✅ 截至 2026-05-12 推荐 |
| OpenClaw Agent | `openclaw_agent` | subprocess openclaw agent | 60s+ | read/exec/memory | 远期可用性改善后推荐 |

### 4.3 可配置触发模式

通过 `TCLAW_TRIGGER_PATTERNS` 环境变量允许用户自定义触发分析的正则模式：

| 预设 | 触发模式 | 适用场景 |
|------|---------|---------|
| errors_only (默认) | ERROR\|FATAL\|CRITICAL\|PANIC\|SEVERE\|Traceback\|Exception | 生产环境监控 |
| warnings_too | 以上 + WARN\|Warning\|failed\|timeout\|denied\|refused\|not found | 开发调试 |
| everything | .* (每行都触发) | 全量分析，配合 LLM AI 过滤 |
